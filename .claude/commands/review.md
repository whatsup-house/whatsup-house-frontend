현재 브랜치에서 변경된 파일을 읽고, 프로젝트 규칙에 따라 코드를 검토한다.

다음 순서로 진행한다:

1. `git diff main --name-only` 또는 `git diff HEAD~1 --name-only`로 변경된 파일 목록을 확인한다.
2. 변경된 파일을 읽는다.
3. 아래 기준으로 검토한다.

---

## 검토 기준

### 1. 데이터 흐름
- 컴포넌트가 `apiClient`나 `lib/api/*.ts`를 직접 import하는지
- React Query 훅 없이 컴포넌트에서 fetch를 직접 호출하는지
- useQuery / useMutation을 컴포넌트 파일 안에 직접 작성했는지 (lib/hooks/로 분리해야 함)

### 2. 타입 안전성
- `any` 타입을 사용했는지
- API 응답 타입이 `lib/api/types.ts`에 정의되지 않고 인라인으로 작성됐는지
- props interface가 빠진 컴포넌트가 있는지

### 3. 상태 관리
- 서버 데이터를 Zustand에 넣었는지
- 클라이언트 상태를 React Query로 관리했는지
- persist 스토어를 SSR 환경에서 hydration 없이 읽었는지

### 4. 컴포넌트 규칙
- `use client`를 불필요하게 추가했는지
- `components/ui/` 안에 도메인 로직이 들어있는지
- page.tsx에 직접 UI 로직이 과도하게 들어있는지

### 5. 스타일링
- 임의 hex 색상을 직접 썼는지 (커스텀 토큰을 써야 함)
- 인라인 style 속성을 사용했는지
