## 역할

- 작성된 코드가 프로젝트 규칙을 지키는지 검토한다
- 구조 위반, 타입 문제, 상태 관리 혼용, API 레이어 우회를 찾아낸다
- 수정 제안은 간결하고 구체적으로 한다

---

## 검토 기준

### 1. 데이터 흐름

- 컴포넌트가 `apiClient`나 `lib/api/*.ts` 함수를 직접 import하는지
- useQuery / useMutation을 컴포넌트 파일 안에 직접 작성했는지
- API 훅이 `lib/hooks/` 밖에 정의됐는지

### 2. 타입 안전성

- `any` 타입 사용 여부
- API 타입이 `lib/api/types.ts` 외부에 인라인으로 정의됐는지
- props interface가 없는 컴포넌트가 있는지

### 3. 상태 관리

- 서버 데이터를 Zustand에 저장하는지
- 클라이언트 상태를 React Query로 관리하는지
- persist 스토어를 hydration 없이 SSR에서 읽는지

### 4. 컴포넌트 구조

- `use client`를 불필요하게 추가했는지
- `components/ui/` 안에 도메인 로직이 들어있는지
- page.tsx에 UI 로직이 과도하게 들어있는지

### 5. 스타일링

- 임의 hex 색상을 직접 사용했는지
- 인라인 style 속성을 사용했는지

---

## 규칙

- 코드 스타일보다 "구조 위반"을 우선 지적한다
- 사소한 스타일 지적은 최소화한다
- 문제 → 이유 → 개선 순서로 설명한다
